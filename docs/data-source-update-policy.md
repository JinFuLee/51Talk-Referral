## 二、资源约束下的优先级队列

### 场景 1：正常运营（资源充足）

**每日更新**（7个数据源）:
1. 口径对比（P0，必须）
2. leads明细（P1，CC排名核心）
3. leads达成（P1，团队对标）
4. 围场汇总（P1，围场分析主体）
5. 课前课后（P1，跟进分析主体）
6. 围场跟进（P1，触达率补充）
7. 打卡率（P1，但可降至每周）

**每周更新**（3个数据源）:
- 当月效率（P2，周一更新）
- 订单明细（P2，周一更新）
- 打卡率（若资源紧张从每日降至每周）

**每月更新**（3个数据源）:
- 月度环比（P2，每月1日）
- 月度同期（P2，每月1日）
- ROI成本（P3，每月5日）

---

### 场景 2：资源紧张（人力不足或数据源临时中断）

#### 最小可行集合（MVC - Minimum Viable Collection）

**必须每日更新**（1个）:
- ✅ **口径对比**（P0）：报告生成的唯一阻塞因素，缺失则系统完全失效

**建议每日更新**（4个，优先级递减）:
- ✅ **leads明细**（P1 #1）：CC排名核心单点，影响个人绩效评估
- ✅ **leads达成**（P1 #2）：团队级对标，管理层每日关注
- ✅ **围场汇总**（P1 #3）：运营每日关注，围场分析主体
- ✅ **课前课后**（P1 #4）：CC跟进质量核心数据

**可延后至每周**（3个）:
- 🟡 围场跟进（P1 #5，降至每周一）
- 🟡 打卡率（P1 #6，已建议每周）
- 🟡 当月效率（P2 #7，降至每周一）

**可延后至按需**（4个）:
- ⚪ 订单明细（P2，ROI已有降级）
- ⚪ 月度环比（P2，可用口径对比替代）
- ⚪ 月度同期（P2，非核心）
- ⚪ ROI成本（P3，降级逻辑成熟）

## 三、时效性要求与业务场景对齐

### 3.1 业务时间线分析

| 业务节点 | 数据需求 | 需要的数据源 | 建议更新频率 | 原因 |
|---------|---------|------------|-------------|------|
| **每周二发布报告**（当前流程） | T-1日数据（周一数据） | 口径对比、leads明细、leads达成、围场汇总、课前课后、围场跟进 | 每日（周一23:59前） | 周二09:00生成报告，需周一23:59前数据到位 |
| **每日运营晨会**（假设场景） | T-1日关键指标 | 口径对比（核心看板）、围场汇总（昨日围场进度） | 每日（T-1日23:59前） | 晨会需昨日数据支持决策 |
| **每周CC绩效评估**（假设场景） | 本周CC个人表现 | leads明细、课前课后、打卡率、围场跟进 | 每周一次（周一09:00前） | 周会评估上周绩效，周一汇总即可 |
| **每月财务对账**（假设场景） | 本月订单明细、ROI成本 | 订单明细、ROI成本 | 每月一次（月初5日前） | 财务月结后核对，月更新足够 |
| **季度战略复盘**（假设场景） | 季度趋势对比 | 月度环比、月度同期 | 每月一次 | 季度复盘基于月度数据，月更新足够 |

**结论**:
- **每周二报告发布**是当前最核心的业务节点，需保证 **7个数据源每日更新**（口径对比 + 6个P1）
- 若业务节奏调整为**双周报告**，P1数据源可降至**每周更新**，资源消耗减半
- **CC绩效评估**若改为月度，打卡率/围场跟进可降至**每月更新**

### 3.2 数据新鲜度要求（T-N定义）

| 数据源 | 建议新鲜度 | 定义 | 监控阈值 | 超阈值影响 |
|-------|-----------|------|---------|----------|
| 口径对比 | **T-1**（昨日数据） | 数据提取时间 ≤ 报告生成时间 - 24小时 | 新鲜度 > 48小时 | 🔴 报告无法生成 |
| P1每日数据源 | **T-1**（昨日数据） | 同上 | 新鲜度 > 72小时 | 🟡 相关章节显示"数据待更新" |
| P1/P2每周数据源 | **T-7**（上周数据） | 数据覆盖上周完整周期（周一至周日） | 新鲜度 > 14天 | 🟢 章节为空，不影响报告整体 |
| P2/P3每月数据源 | **T-30**（上月数据） | 数据覆盖上月完整月份（1日至月末） | 新鲜度 > 60天 | 🟢 使用降级逻辑 |

**监控建议**:
- 在 `src/multi_source_loader.py` 中增加 `check_data_freshness()` 方法
- 每次加载数据源时检查文件修改时间（`stat().st_mtime`）
- 超过阈值时在报告中自动标注 "⚠️ 数据新鲜度：XX天前"
- 高级别预警（P0/P1超48小时）自动发送通知

## 五、数据源健康度监控面板建议

### 5.1 监控指标

| 指标类别 | 监控内容 | 阈值标准 | 预警级别 | 通知方式 |
|---------|---------|---------|---------|---------|
| **新鲜度** | 文件最后修改时间 | P0: <48h<br/>P1: <72h<br/>P2: <14d | 🔴/🟡/🟢 | Line Bot + 邮件 |
| **完整性** | 文件大小 > 最小阈值 | >10KB（口径对比）<br/>>5KB（其他） | 🟡 | Line Bot |
| **行数** | Excel行数 > 最小阈值 | >10行（口径对比）<br/>>5行（其他） | 🟡 | Line Bot |
| **数据覆盖** | 必需列是否存在 | 核心列缺失 | 🔴 | Line Bot + 邮件 |
| **更新频率** | 实际更新频率 vs 建议频率 | 连续2次未更新 | 🟡 | Line Bot |

    """数据源健康度检查器"""

    THRESHOLDS = {
        "口径对比": {"max_age_hours": 48, "min_size_kb": 10, "min_rows": 10, "priority": "P0"},
        "leads明细": {"max_age_hours": 72, "min_size_kb": 5, "min_rows": 5, "priority": "P1"},
        # ... 其他数据源
    }

    def check_freshness(self, data_source_name: str, file_path: Path) -> dict:
        """检查数据新鲜度"""
        if not file_path.exists():
            return {"status": "🔴", "message": f"{data_source_name} 文件不存在"}

        threshold = self.THRESHOLDS.get(data_source_name, {})
        max_age = threshold.get("max_age_hours", 168)  # 默认7天

        mtime = datetime.fromtimestamp(file_path.stat().st_mtime)
        age_hours = (datetime.now() - mtime).total_seconds() / 3600

        if age_hours > max_age:
            return {
                "status": "🔴" if threshold["priority"] == "P0" else "🟡",
                "message": f"{data_source_name} 数据过期（{age_hours:.1f}小时前，阈值{max_age}h）"
            }

        return {"status": "🟢", "message": f"{data_source_name} 数据新鲜（{age_hours:.1f}h前）"}

    def check_completeness(self, data_source_name: str, file_path: Path) -> dict:
        """检查数据完整性"""
        threshold = self.THRESHOLDS.get(data_source_name, {})
        min_size_kb = threshold.get("min_size_kb", 5)
        min_rows = threshold.get("min_rows", 5)

        # 检查文件大小
        size_kb = file_path.stat().st_size / 1024
        if size_kb < min_size_kb:
            return {"status": "🟡", "message": f"{data_source_name} 文件过小（{size_kb:.1f}KB，阈值{min_size_kb}KB）"}

        # 检查行数（Excel）
        try:
            df = pd.read_excel(file_path, nrows=min_rows + 1)
            if len(df) < min_rows:
                return {"status": "🟡", "message": f"{data_source_name} 行数不足（{len(df)}行，阈值{min_rows}行）"}
        except Exception as e:
            return {"status": "🔴", "message": f"{data_source_name} 文件读取失败: {e}"}

        return {"status": "🟢", "message": f"{data_source_name} 完整性检查通过"}

## 六、成本与资源分析

### 6.1 人力成本估算

| 更新频率策略 | 每周人工投入（小时） | 每月人工投入（小时） | 适用场景 |
|------------|------------------|------------------|---------|
| **方案A: 每日更新**（7个每日数据源） | 7h（每日1h × 7天） | 28h | 资源充足，业务需要每日/每周报告 |
| **方案B: 双周报告**（7个每周数据源） | 2h（周一集中2h） | 8h | 资源紧张，业务可接受双周报告 |
| **方案C: 月度报告**（7个每月数据源） | 0.5h | 2h | 极度资源紧张，仅需月度复盘 |
| **最小可行集合（MVC）**（口径对比 + 4个P1） | 5h | 20h | 保证核心功能，牺牲部分分析维度 |

**假设**:
- 每个数据源手工更新平均耗时 **15分钟**（下载 → 整理 → 上传 → 验证）
- 每日7个数据源 = 7 × 15分钟 = **1.75小时/天**
- 自动化后（L2半自动）可减少至 **5分钟/数据源**

### 6.2 自动化ROI分析

| 数据源 | 手工耗时（分钟/次） | 更新频率 | 月耗时（小时） | 自动化后月耗时（小时） | 节省时间 | 开发投入 | ROI（回本周期） |

## 八、关键决策点

### 决策点 1: 报告发布频率

**选项**:
- A. 保持每日生成报告（当前 `cron: 09:00`）
- B. 改为每周二生成报告（`cron_weekday: 2`）
- C. 改为双周/月度生成报告

**建议**: **选项 B（每周二生成报告）**
**理由**:
- 当前数据源已支持每日更新，但报告消费频率未知（假设为每周）
- 若业务团队仅每周查看报告，每日生成是资源浪费
- 每周生成可将数据源更新从每日降至每周，节省 **72%人力**（7天 → 1天）

**需要您决策**:
请确认报告实际查看频率（每日/每周/双周/月度），以便调整调度策略。

### 决策点 2: 资源约束下的取舍

**问题**: 若数据源更新人力不足，如何取舍？

**选项**:
- A. 保持7个每日数据源（口径对比 + 6个P1），牺牲其他工作时间
- B. 缩减至MVC最小集合（口径对比 + 4个核心P1），其他P1降至每周
- C. 投入1个月开发leads明细自动化，长期节省9.5h/月

**建议**: **选项 C（投入自动化）**
**理由**:
- 短期痛苦（1周开发）换取长期收益（9.5h/月 × 12月 = 114h/年）
