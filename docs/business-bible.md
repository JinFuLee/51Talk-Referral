### 1.2 三大岗位定义

| 代码 | 全称 | 职责 | 数据别名 |
|------|------|------|---------|

> **铁律**：面板/报告统一使用 **CC / SS / LP**，遇到 EA 自动映射为 SS，遇到 CM 自动映射为 LP。数据中 "-" 占位符团队映射为 THCC。

## 第 2 章：转介绍业务模型 {#ch2-referral}

> 来源：`docs/glossary.md` + `docs/bi-indicator-dictionary.md §1` + `projects/referral/config.json`（`enclosure_role_assignment` / `enclosure_role_narrow` / `enclosure_role_wide` / `channel_metric_scope`）

### 2.1 转介绍定义

一条注册被认定为"转介绍"需满足以下**任一**条件：
1. 一级渠道 = "转介绍"
2. 一级渠道 ≠ 转介绍渠道，但有推荐人 ID，且推荐人 ID ≠ 0

### 2.2 渠道分类与 SQL 映射

```sql
CASE
  WHEN chnl4_name IN ('cc_recommend', 'dapan_chai_zhanghao')
    THEN 'CC窄口径'
  WHEN chnl4_name IN ('is_recommend')
    THEN 'SS窄口径'
  WHEN chnl4_name IN ('lp_recommend', 'lp_recommend.', 'dapan_ss_crm_haibao')
    THEN 'LP窄口径'
  ELSE '宽口径'
END
```

> **注意**：`lp_recommend.`（含句号）疑为数据源脏数据，已在 BI 侧确认保留原样映射至 LP 窄口径。

| 渠道 | 描述 | 质量 | 付费转化率参考 |
|------|------|------|--------------|
| CC窄口径 | CC 员工链接绑定（主动推荐） | 高 | ~32% |
| SS窄口径 | SS(EA)员工链接绑定（SOP 向用户要推荐码） | 中 | ~28% |
| LP窄口径 | LP(CM)员工链接绑定（另有"同言同语"宽口活动） | 中 | — |
| 宽口径 | 老学员（UserA）自主分享链接，不区分部门 | 低 | — |

### 2.3 围场分段定义

围场 = 用户**付费当日**起算的天数分段（付费当日为 Day 0）。

#### 天数制围场（精确，用于日常运营分析）

| 围场代码 | 天数范围 | 月份对应 | 典型参与率 | 典型打卡率 |
|---------|---------|---------|-----------|-----------|
| d0_30 | 付费后 0-30 天 | M0（付费当月） | **22.4%** | **96.5%** |
| d31_60 | 付费后 31-60 天 | M1 | 10.5% | 44.0% |
| d61_90 | 付费后 61-90 天 | M2 | 6.9% | 35.0% |
| d91_120 | 付费后 91-120 天 | M3 | 3.8%（M3-M5均值） | 26.4%（M3-M5均值） |
| d121_180 | 付费后 121-180 天 | M4-M5 | 3.8% | 26.4% |
| d181_plus | 付费后 181+ 天 | M6+ | **2.4%** | **19.2%** |

**关键洞察**：M0 是转介绍黄金期，参与率随时间快速衰减（M0 = 22.4% → M6+ = 2.4%），M0 带新系数 1.35，M1 最高 1.47。

### 2.4 围场×岗位责任矩阵

#### 窄口径（员工主动联系学员）

| 围场（天数） | 围场（月份） | 负责岗位 | 指标口径 |
|------------|------------|---------|---------|
| 0-90 天 | M0-M2 | **CC** | full_funnel（注册/预约/出席/付费/金额+全转化率） |
| 91-120 天 | M3 | **SS** | leads + 过程指标（触达率/打卡率）+ 跨岗 leads→CC 转化率 |
| 121 天+ | M4+ | **LP** | leads + 过程指标（触达率/打卡率）+ 跨岗 leads→CC 转化率 |

#### 宽口径（学员自主分享场景）

| 围场（天数） | 围场（月份） | 负责岗位 |
|------------|------------|---------|
| 0-90 天 | M0-M2 | **CC** |
| 91-120 天 | M3 | **SS** |
| 121-180 天 | M4-M5 | **LP** |
| 181 天+ | M6+ | **运营** |

> **配置权威来源**：`projects/referral/config.json` → `enclosure_role_narrow`（窄口径）/ `enclosure_role_wide`（宽口径）。Settings 页面写入，前端 `useWideConfig()` 从 API `/api/config/enclosure-role` 读取，禁止硬编码。

### 2.5 口径×指标归属规则

| 口径 | 角色 | 可用指标范围 |
|------|------|------------|
| CC（full_funnel） | CC | 注册→预约→出席→付费→金额，全部转化率（全漏斗） |
| SS/LP（leads_and_process） | SS、LP | 转介绍 leads 数 + 过程指标（触达率/打卡率）+ leads→CC 转化率（跨岗效率，非自身 KPI） |
| 宽口径（leads_only） | 宽口径 | 仅转介绍 leads 数 |

**特别说明**：SS/LP 的"转化率"= SS/LP 带来的 leads 被 CC 转化为付费的比率（跨岗效率参考），不是 SS/LP 自身的销售漏斗转化。

### 2.6 有效学员定义

有效学员（可参与转介绍的基础池）须同时满足：
1. **首大单付费**：完成过首次大额付费
2. **次卡 > 0**：持有课时卡余额
3. **套餐在有效期内**：未过期
4. **账号状态 = 激活（on）**：账号正常使用中

### 2.7 学员构成分类

| 分类代码 | 全称 | 说明 |
|---------|------|------|
| aa | 美小 | 美式英语小班课学员 |
| k12 | 青少 | 青少年学员 |
| non-k12 | 成人 | 成人学员 |

泰国与全球分类标准一致。

### 2.8 SS/LP 转介绍参与说明

- **SS 窄口径**：SS 主动向用户要推荐码（操作 SOP 简单，直接请求用户提供推荐码）
- **LP 宽口径**："同言同语"活动——用户在 APP 完成跟读后分享到社群打卡，数据暂存于另一个 BI 系统，待 API 接入后集成

### 2.9 转介绍公式

```
转介绍产出 = 活跃用户 × 参与率 × 获客率 × 转化率
```

**演化阶段**：基础启动（激励驱动）→ 科学运营（公式化）→ 系统思维（两大存量经营）

## 第 3 章：KPI 指标字典 {#ch3-kpi}

> 来源：`docs/bi-indicator-dictionary.md` + `docs/glossary.md` + `docs/research/key-metrics-quick-reference.md`

### 3.1 学员层指标

**有效通话标准统一为 >=20s**（触达率/通时通次均以此为准，已从原文件 120s 更正）

| 指标名 | 公式 | 单位 | 备注 |
|-------|------|------|------|
| **触达率** | 当月有效接通学员数 / 当月有效学员数 | % | 有效接通 = 通话时长 **≥20s** |
| **参与率** | 当月带来 ≥1 个转介绍注册的学员数 / 当月有效学员数 | % | 衡量转介绍活跃度 |
| **打卡率** | 当月转码且分享的学员数 / 当月有效学员数 | % | 转码 = 生成/激活转介绍专属二维码或链接 |
| **带新系数** | 当月 B 学员注册人数 / 当月带来注册的 A 学员人数 | 倍 | 每个推荐人平均带来几个新注册 |
| **带货比** | 当月推荐注册人数 / 当月有效学员数 | % | 整体池子产出效率 |
| **带新注册数** | 当月 A 学员带来的 B 注册人数 | 人 | 绝对值 |
| **拨打率** | 当月有 CC 拨打学员数 / 月底有效学员数 | % | 过程跟进覆盖率 |

### 3.2 漏斗指标

```
注册 → 预约 → 出席 → 付费
```

| 指标名 | 公式 | 单位 | 备注 |
|-------|------|------|------|
| **注册（Leads）** | B 学员通过转介绍链接注册数 | 人 | 漏斗入口 |
| **预约数** | 预约体验课人数 | 人 | 首次预约，不计多次 |
| **出席数** | 出席体验课人数 | 人 | 首次出席，不计多次 |
| **付费数** | 完成付费的人数 | 人 | 去小单口径：签约金额 ≥1500 USD（CSCC ≥1400）|
| **预约率** | 预约人数 / 注册人数 | % | Appt% |
| **预约出席率** | 出席人数 / 预约人数 | % | Show up% |
| **出席付费率** | 付费人数 / 出席人数 | % | Show up to pay% |
| **注册付费率** | 付费人数 / 注册人数 | % | 端到端转化率 |
| **客单价（ASP）** | 付费金额(USD) / 付费单量 | USD | 大单口径 ≥1500 USD |

**2026-01 样本数据**：总注册 582，预约 440，出席 266，付费 113，金额 111,636 USD；预约率 75.6%，预约出席率 60.5%，出席付费率 42.5%，整体注册付费率 19.4%。

### 3.3 基础达成

**本系统业绩 = 签约金额（去小单），不含激励虚拟 / 判单虚拟 / 退款**

| 指标名 | 公式 / 定义 | 单位 |
|-------|------------|------|
| Target | CC 当月基础标 | USD |
| 金额（毛业绩） | MTD CC 签约金额 + 切割业绩金额 | USD |
| 去小单付费金额 | 大单（签约金额 ≥1500；CSCC ≥1400）签约金额 | USD |
| 激励虚拟业绩 | MTD CC 激励虚拟业绩汇总 | USD |
| 判单虚拟业绩 | MTD CC 判单虚业绩汇总 | USD |
| 退款金额 | MTD CC 退款金额汇总 | USD |
| **净业绩** | 金额 + 激励虚拟业绩 + 判单虚拟业绩 - 退款金额 | USD |
| 完成率 | 净业绩 / Target | % |
| 业绩是否达标 | if(完成率 ≥ BM, 1, 0) | 布尔 |
| 达标人数占比 / 业绩达标率 | SUM(业绩是否达标) / SUM(CC个人) | % |
| 平均客单价(>1500) | 去小单付费金额 / 去小单付费数 | USD |

### 4.2 SS/LP 排名：四类 5 维

**综合公式**：`composite_score = process×0.25 + result×0.30 + quality×0.25 + contribution×0.20`，满分 1.0
**归一化规则**：SS 和 LP 分别独立归一化，不混合排名

#### 过程指标（25%）

| 指标 | 权重 | 数据源 |
|------|------|--------|
| 触达率（contact_rate） | 12.5% | outreach.by_cc (F5) |
| 打卡率（checkin_rate） | 12.5% | kpi.north_star_24h (D1) / kpi.checkin_rate_monthly (D5) |

#### 结果指标（30%）

| 指标 | 权重 | 数据源 |
|------|------|--------|
| 转介绍用户数（leads） | 30% | leads.personal (A4) |

#### 质量指标（25%）

| 指标 | 权重 | 数据源 |
|------|------|--------|
| leads→CC 转化率 | 25% | leads.personal (A4) paid/leads |

#### 贡献指标（20%）

| 指标 | 权重 | 数据源 |
|------|------|--------|
| 业绩贡献占比（paid_share） | 20% | 个人 CC 转化付费数 / 同角色团队总 CC 转化付费数 |

### 4.3 关键算法细节

| 机制 | 说明 |
|------|------|
| **min-max 归一化** | (v - min) / (max - min)，范围 [0, 1]；所有人值相同时归一化为 0.5 |
| **_redistribute** | CC：数据缺失时权重等比分摊到同类有数据的其他维度，保证类别得分满分仍为 1.0 |
| **_redistribute_role** | SS/LP：同上，在同角色内分摊 |
| **paid_share** | SS/LP 贡献指标 = 个人 CC 转化付费数 / 同角色团队总 CC 转化付费数 |
| **leads_to_cc_rate** | SS/LP 质量指标 = A4 字段 `conversion_rate`，或 paid/leads 计算，与 CC 转化率同口径 |

### 4.4 CC vs SS/LP 区别对比

| 维度 | CC（3类 18维） | SS/LP（4类 5维） |
|------|--------------|----------------|
| 过程 | 7 维（外呼/接通/有效接通/四种跟进） | 2 维（触达率/打卡率） |
| 结果 | 7 维（注册/leads/转介绍用户/客单价/单量/业绩/占比） | 1 维（leads 数） |
| 效率 | 4 维（转化率/打卡率/参与率/带新系数） | 无独立效率类 |
| 质量 | 无独立质量类 | 1 维（leads→CC 转化率） |
| 贡献 | 业绩占比归入结果类 | 独立贡献类（paid_share） |
| 总权重 | process×0.25 + result×0.60 + efficiency×0.15 | process×0.25 + result×0.30 + quality×0.25 + contribution×0.20 |
| 归一化范围 | 所有 CC 中归一化 | SS 和 LP 各自独立归一化 |

### 5.2 双差额体系

每个数值 KPI 必须计算**两种差额 + 两个日均**：

| 差额类型 | 公式 | 含义 |
|---------|------|------|
| **目标绝对差** | `actual - target` | 距月目标差额（负=落后，正=超额） |
| **时间进度差** | `actual/target - time_progress` | 是否跟上当前时间进度（负=掉队） |
| **达标需日均** | `(target - actual) / remaining_workdays` | 完成月目标每天需要多少 |
| **追进度需日均** | `max(0, target × time_progress - actual) / remaining_workdays` | 追上时间进度线每天需要多少 |

**后端字段对应**：`absolute_gap`（目标绝对差）、`gap`（进度差）、`remaining_daily_avg`（达标需日均）、`pace_daily_needed`（追进度需日均）

### 5.3 状态标签

| 标签 | 进度差阈值 | 颜色 |
|------|----------|------|
| 🟢 持平/超额 | gap ≥ 0% | 绿色 |
| 🟡 落后 | -5% ≤ gap < 0% | 黄色 |
| 🔴 严重落后 | gap < -5% | 红色 |

**配置来源**：`projects/referral/config.json` → `gap_thresholds`（`green: 0.0, yellow: -0.05`）

### 8.2 三类 ROI 公式

```python
# 次卡 ROI

# 综合 ROI
综合ROI = 实际营收 / (次卡成本 + 现金成本)
```

### 10.3 转介绍三阶段演化模型

| 阶段 | 核心驱动 | 关键特征与工具 |
|------|---------|--------------|
| **1. 基础启动** | 用户意愿（激励为主） | 结果激励（60美金/20节课）、工具能力（VIP/友谊卡/小精灵/推荐码）、四大红利（市场/启动/存量/销售） |
| **2. 科学运营** | 公式化运营 | 转介绍公式 = 活跃用户 × 参与率 × 获客率 × 转化率；多渠道精细化（手拿嘴要/打卡活动/运营直播/社群运营/合伙人/进校合作） |
| **3. 系统思维** | 两大存量经营 | 存量一：活跃满意用户 = 产品质量 + 服务体验 = 用户满意；存量二：用户人脉池（高信任高需求/高信任低需求/低信任高需求） |

### 10.4 转介绍运营因果模型

```
01 业务驱动（因）           02 用户感知（根基）         03 层级跃迁（果）
├─ 系统：精准度+工作台      ├─ 意愿：外部策略+奖励      全量付费用户
├─ 用户服务：海报+文案+奖品  ├─ 能力：业务能力+一键分享     ↓ 参与转介绍的用户
├─ 人：CC想不想做/会不会做  ├─ 环境：场景多元化              ↓ 活跃转介绍的用户
└─ 策略：政策支持           └─ 产品：操作简化                   ↓ 合伙人
```

**关键杠杆**：人（CC 的意愿与能力）> 产品（操作简化）> 策略（政策激励）。

### 10.5 金字塔 5-Why 诊断法

适用场景：指标偏离 >2σ 时自动触发

```
结果异常（如参与率 -30%）
  → Why 1: 打卡率下降？（数据支撑：打卡率 70% → 55%）
    → Why 2: 打卡活动奖励减少？（核实激励政策变动）
      → Why 3: 预算削减？（确认成本政策）
        → Why 4: ROI 未达标导致？（核实 ROI 数据）
          → Why 5: 转化率下降拉低 ROI（根因）
            → 行动：优化 M0 触达 SOP + 提升出席付费率
            → 预期：参与率 +1.5pp，带新 +45人/月，业绩 +$38K/月
```
