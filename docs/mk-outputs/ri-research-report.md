## 四、WebSearch 调研结论

### 1. openpyxl read_only 模式 vs pandas（D4 10002行场景）

**结论**：推荐 `pandas.read_excel()` + `openpyxl` 引擎组合，不推荐纯 openpyxl。

- **openpyxl read_only**：流式读取，内存占用极低（50MB文件仅用几MB），但返回的是行迭代器，聚合运算需手动写循环，代码量大。runtime从20分钟→2秒的优化案例适合只需逐行扫描的场景。
- **pandas.read_excel(engine='openpyxl')**：底层自动使用openpyxl，对10002行×59列的D4完全无压力（约6MB数据），pandas向量化运算远快于Python循环，且支持groupby直接聚合渠道金额。
- **推荐方案**：D4/D2用 `pd.read_excel(path, engine='openpyxl')` 读取（~100ms），D1/D3/D5行数少可用 openpyxl read_only。不需要引入 DuckDB/Parquet（当前规模不值得复杂度）。
- 来源：[openpyxl 优化模式文档](https://openpyxl.readthedocs.io/en/stable/optimized.html)、[性能文档](https://openpyxl.readthedocs.io/en/stable/performance.html)

### 2. Next.js App Router 大数据表格方案（D4 10002行 × 59列）

**结论**：**服务端分页**优先，D4全量前端渲染禁止。

- **虚拟滚动**（TanStack Virtual）：适合行数 ≤50000、用户需连续滚动的场景，但 10002行 × 59列数据仍需全量传输到客户端（~5MB JSON），初始加载慢。
- **服务端分页**（TanStack Table + Next.js URL searchParams）：推荐方案。FastAPI返回 `{data: rows[], total: N, page: P, pageSize: 50}`，前端通过 `?page=1&pageSize=50` 参数请求。每次请求仅传50行×59列（~15KB），性能最优。
- **实现路径**：`/api/d4/students?page=1&pageSize=50&cc=xxx&enclosure=0~30` → FastAPI用pandas切片，支持按CC/围场过滤。前端已有 shadcn/ui DataTable 可直接接入。
- 来源：[TanStack Table 分页指南](https://tanstack.com/table/v8/docs/guide/pagination)、[Next.js App Router 服务端分页](https://next-app-gamma-lovat.vercel.app/features/pagination/server-side)

### 3. FastAPI Excel 文件监控热更新方案

**结论**：**watchdog** 是标准方案，搭配 pandas 重新读取机制，polling 仅作降级备选。

- **watchdog + FileSystemEventHandler**：监听 `event.src_path.endswith('.xlsx')`，检测到文件修改后触发缓存失效 + 重新读取。主流用法，已有完整 XLSX 示例（[GitHub Gist](https://gist.github.com/d45507fe68b2e48f23ab1150377cf552)）。
- **注意事项**：Excel 写入期间可能触发 PermissionError，需加重试循环（sleep 0.5s × 3次）；macOS 的 watchdog 使用 FSEvents 机制，比 Linux inotify 稍有延迟（~1s）。
- **polling 降级**：`watchdog.observers.polling.PollingObserver` 兼容 Docker volume mount 场景，延迟更高（默认1s检查间隔）但稳定。
- **推荐集成**：FastAPI startup 事件中启动 watchdog Observer，修改触发后设 `_cache_dirty = True`，下次 API 请求时重新加载文件并更新内存缓存。
- 来源：[Pandas + watchdog 实时监控](https://medium.com/@connect.hashblock/how-i-built-a-real-time-data-monitor-using-pandas-and-file-watchers-beab7d3df56f)、[watchdog XLSX Python脚本](https://gist.github.com/d45507fe68b2e48f23ab1150377cf552)

| R5 | D2.是否有效 字段过滤：若直接使用全量D2数据（含"否"行），参与率/带新系数计算会被稀释，产生偏差 | 中 | 加载D2时强制 `df = df[df['是否有效'] == '是']` 过滤，写入数据加载注释文档 |

## 六、结论与建议

1. **10个模块中9个可从5个数据源直接覆盖**，仅模块5/9的渠道金额需聚合推算（D3末次归属×金额）
2. **改造复杂度低于预期**：旧代码的双差额计算、漏斗可视化、SlideShell框架均高度可复用，核心改动集中在数据加载层（新的 5个Excel loader替代旧35个）
3. **技术栈不变**：FastAPI + Next.js + pandas + watchdog 已是项目现有技术栈，无需引入新依赖
4. **优先级建议**：先实现 D1+D2+D3 三源（覆盖模块1/2/3/7/8/10），D4/D5作为第二阶段（模块4/5/6/9依赖渠道金额推算，复杂度高）
