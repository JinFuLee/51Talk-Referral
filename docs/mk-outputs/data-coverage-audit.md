## 建议（按 ROI 排序）

### 建议 1 — 围场矩阵补全 SS/LP 触达率 + 付费结果列

**Before：** 围场×CC 矩阵展示 8 列，SS触达率/LP触达率/付费数/业绩USD 不可见。后端 `EnclosureCCMetrics` 模型已包含这 4 个字段并已返回，前端表头未加列。

**After：** 围场矩阵增加 4 列（SS触达率、LP触达率、付费数、业绩USD），列数从 8 列升至 12 列，SS/LP 围场岗位边界的核心考核指标全部可见。

**ROI：** 纯前端改动 `frontend/app/enclosure/page.tsx`，表头+数据行各加 4 个 `<th>/<td>`，改动量 <20 行，0 后端成本。解锁当前完全不可见的 SS/LP 岗位考核维度。

---

### 建议 2 — 团队卡片补充业绩 + 打卡率

**Before：** CC 团队卡片底部 3 格只展示"参与率/注册数/付费数"，`revenue_usd`/`checkin_rate`/`cc_reach_rate` 已在 `/api/team/summary` 返回但未展示。

**After：** 卡片底部扩展为 5-6 格，加入业绩（USD）、打卡率，使每张卡片同时呈现"结果（注册/付费/业绩）+ 过程（参与率/打卡率）"完整画像。

**ROI：** 纯前端改动 `frontend/app/team/page.tsx`，卡片 grid 扩列，改动量 <30 行，0 后端成本。日会中对 CC 的评估从"注册+付费"升级为"全维度"。

---

### 建议 3 — 首页总览加达成率 Badge

**Before：** 首页 6 个 KPI 卡片只展示绝对值，`区域单量达成率`/`区域业绩达成率`/`区域转介绍客单价达成率` 3 个字段在 `metrics` 中已存在，但 `KPI_CARDS` 数组未包含。

**After：** KPI 区增加 3 个达成率卡片，或在现有业绩/单量/客单价卡片右上角附加达成率 Badge（绿色>=100%/黄色>=80%/红色<80%）。管理层打开首页第一眼看到达标情况。

**ROI：** 纯前端改动 `frontend/app/page.tsx`，`KPI_CARDS` 添加 3 项或 StatCard 加 badge prop，改动量 <40 行，0 后端成本。达成率是管理层最高频需求，当前完全看不到。

---

### 建议 4 — 接入 TimePeriod，展示追进度需日均

**Before：** 所有指标均为月度累计绝对值，无时间进度参照。`backend/core/time_period.py` 已存在但从未被任何 API 调用。运营不知道当前速度是否能完成月目标。

**After：** `overview.py` 和 `funnel.py` 集成 `TimePeriod`，为每个目标指标额外返回 `time_progress`（时间进度%）、`remaining_daily_avg`（达标需日均）、`pace_daily_needed`（追进度需日均）。首页新增"今天是月的第 X 工作日，当前进度 Y%，完成月目标每天还需 Z 单"提示条。

**ROI：** 后端 `backend/api/overview.py` + `backend/api/funnel.py` 各约 20 行，前端新增 1 个进度条组件约 50 行。将静态报表变成实时决策工具，运营日会的核心输入从"看数字"变成"看差距"。

---

### 建议 5 — D3 接通列接口化，补全触达质量分析

**Before：** D3 明细（560行×19列）的 `CC接通`/`SS接通`/`LP接通`/`有效打卡` 列完全未被任何 API 消费，D3 整体利用率仅 11%。

**After：** 新增 `/api/detail/reach-stats` 接口，按 CC/SS/LP 聚合有效接通次数和打卡人数。在围场分析或团队页增加"触达质量"子 Tab，区分"触达了（触达率）"和"有效触达（接通率）"两个层次。

**ROI：** 后端新增 1 个 API 约 40 行，前端新增 1 个 Tab 约 60 行。D3 是唯一有学员级接通记录的源，当前整张表几乎被完全浪费。接通率分析是判断触达质量的关键维度。

## 汇总评分

| 维度 | 评分 | 说明 |
|---|---|---|
| 数据加载完整性 | 5/5 | D1-D5 全部成功接入，无遗漏文件 |
| 后端字段利用率 | 3/5 | D1(100%)、D2(76%)、D3(11%)、D4(34%)、D5(79%)；加权约 **57%** |
| 前端展示覆盖率 | 2/5 | 大量后端已计算字段前端未展示；D2 围场 API 返回 14 字段但前端只用 8 个 |
| 运营决策支持度 | 3/5 | 漏斗+围场+高潜骨架已成，缺时间趋势+触达质量+达成率可视化 |
| **综合覆盖率** | **~60%** | 瓶颈是"有数据但看不到"，而非"没有数据" |

> 核心结论：建议 1-3 均为纯前端改动（每项 <1 小时），零后端成本，可立即将覆盖率从 60% 提升至约 80%。建议 4-5 需少量后端工作，解锁时间进度和触达质量两个当前完全缺失的运营维度。
