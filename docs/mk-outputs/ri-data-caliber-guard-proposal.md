# 数据口径守卫方案（待评审）

## 背景

ref-ops-engine 的所有数据指标必须是**转介绍口径**（非总销售业绩）。当前数据正确靠：
1. Quick BI 表本身列名含"转介绍"
2. 后端 regex `^TH-CC\w+Team$` 过滤团队
3. 无自动化检测

## 已验证的当前数据

| 数据源 | 指标 | 值 | 口径确认 |
|--------|------|-----|---------|
| D1 泰国汇总 | revenue | $199,372 | 列名"总带新付费金额USD" |
| D1 泰国汇总 | leads | 1,138 | 列名"转介绍注册数" |
| D1 泰国汇总 | paid | 209 | 列名"转介绍付费数" |
| D2→API 聚合 | revenue | $196,435 | 过滤后 7 团队 109 CC |
| D2→API 聚合 | leads | 1,010 | 同上 |
| D2→API 聚合 | paid | 205 | 同上 |
| D1 vs API 差额 | revenue | $2,937 (1.5%) | D1 含非 CC 销售团队行 |

## 风险场景

1. **BI 表结构变更** — 列名改了/新增了非转介绍列，后端静默读错数据
2. **非转介绍数据混入** — 如果 BI 端加入总销售列，系统可能错误消费
3. **团队 Regex 不匹配** — BI 端新增团队命名不符合 pattern，数据静默丢失或混入
4. **D1 与 D2 口径不一致** — 两张表理应都是转介绍，但无交叉验证

## 提议的方案：3 层自动校验

### 层 1 — 列名契约校验
加载每个 Excel 后，验证 critical_columns 全部存在且含"转介绍"关键词的列确实存在。
缺列 → 启动告警（钉钉 + error-log.jsonl），阻止该数据源进入计算。

### 层 2 — D1 vs D2 交叉校验
D2 聚合后的 revenue/leads/paid 必须满足：
- D2_filtered ≤ D1_泰国（上界：不能超过区域总量）
- D2_filtered ≥ D1_泰国 × 0.80（下界：过滤不能太激进）

违反上界 → 非转介绍数据混入告警（P0）
违反下界 → 过滤过度告警（P1），输出被排除团队清单

### 层 3 — 团队过滤覆盖率
每次 _agg_d2 执行后记录：
- kept_teams / total_teams / excluded_teams_with_revenue
- excluded 中有 revenue > 0 的非 TH-CC 团队 → advisory 日志
- kept_teams 数量 vs 上次变化 ≥2 → 团队变动告警

### 实现位置

| 层 | 文件 | 触发时机 |
|----|------|---------|
| 1 | `backend/core/data_manager.py` load_all() | 数据加载时 |
| 2 | `backend/api/cc_performance.py` 新增 _cross_validate() | API 被调用时 |
| 3 | `backend/api/cc_performance.py` _agg_d2() 末尾 | 每次聚合后 |

告警输出统一写入 `output/data-caliber-audit.jsonl`，Dashboard 数据源状态卡片消费。

### ROI
- 投入：~100 行 Python，不改数据流，只加校验
- 产出：BI 端任何变更 → 系统自动发现，不依赖人肉比对
- 风险：零——只读校验，不改计算逻辑
