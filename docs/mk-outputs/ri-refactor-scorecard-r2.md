## 改进目标（≤3 条，按 ROI 排序）

### 改进 1（优先，预计 +2 分，弥补达标缺口）

**对象：SEE 合规 §子项A 全局扫描 + 科学理论 D1链乘阈值**

- Before：旧 API 依赖 Grep 扫描命令给出未执行（零实际命中列表）；D1 链乘 ±2% 来源仅标注"经验值"无推导路径
- After：① 执行 `grep -rn "from.*api.*import\|/api/" frontend/src --include="*.ts" --include="*.tsx"` 并将命中结果列入报告（Phase 1 扫描结果前置到评审层）；② 为 ±2% 补充推导依据（如"D1 仅 2 行，单期数据浮动参考 TanStack 分页误差范围，实测后 14 天内校准"），或将阈值调整为可推导的值（±5% 对齐 D3 末次归属容忍标准，统一量级）
- ROI：投入 0.5 人天 → 总分从 101 跨过 102 达标线，同时消除 Phase 1 执行风险盲区

### 改进 2（次优，预计 +2 分）

**对象：可量化 — 旧 Loader 代码量 Before 基线**

- Before：旧 Loader 总行数仅参考 ops_loader.py ~200 行，未从代码库实际统计，"删除 80% 旧代码"量化断链
- After：执行 `wc -l backend/core/loaders/*.py` 统计现有所有 Loader 文件行数，将结果写入报告；同时补充 funnel_detail.py 行数（ScenarioEngine 复用 5-6 人天的估算依据）
- ROI：投入 0.25 人天 → 可量化维度从 16 升至 17-18，增强方案可信度，同时为 Phase 4 清理提供可验证 Before 基线

### 改进 3（可选，预计 +1 分）

**对象：可溯源 — GitHub Gist 来源级别修正**

- Before：watchdog 方案引用 GitHub Gist 标注为 A 级来源，但 Gist 为个人代码片段
- After：将来源级别修正为 C 级，并补充官方 watchdog 文档（[watchdog.readthedocs.io](https://watchdog.readthedocs.io/)，A 级）作为主引用，Gist 作为实现参考
- ROI：投入 0.1 人天 → 来源分级合规，报告学术严谨性提升
